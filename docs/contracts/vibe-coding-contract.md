# Vibe Coding 执行契约

`aicore` 的目标不是替代安全审计、专项性能分析或全局架构委员会，而是把 Vibe Coding 从“想到哪改到哪”收束成“先做任务级架构审核，再动手、可追溯、可回退”的执行流程。

这份契约用于约束 Agent 和人类协作时的最小行为边界，减少需求飘移、实现发散、命名混乱、半重构、重复实现、风格失控等问题。

## 1. 定位

- `aicore` 不是安全扫描器
- `aicore` 不是数据库优化器
- `aicore` 不替代专项架构审计工具
- `aicore` 是任务级架构闸机、改动留痕器、完成态账本

所以它主要防的是：

- 任务范围失控
- Agent 私自脑补场景
- 主入口不清晰
- 模块职责和实现入口不清晰
- 修改范围失控
- 接口、状态、命名、分页等基础语义没有先冻结
- 验收依据缺失
- 改动做到一半就宣称完成
- 事实、历史、账本混在一起

## 2. 默认流程

每个新任务默认按下面顺序走：

1. `start`：先生成任务草案
2. `update`：补全任务级架构信息、修改范围、验收依据、风险、回退方案
3. `approve`：冻结当前任务边界
4. 实现代码
5. `log-write`：记录本轮改动
6. `checkpoint`：在稳定点保存阶段状态
7. `ledger-confirm`：只有确认成为当前系统事实时才落账

任何跳过 `start / update / approve` 直接进入实现的行为，都视为违规。

## 3. 任务级架构契约

### 3.1 单一模块

- 一个任务只能解决一个稳定模块的问题
- 如果请求同时涉及认证、支付、存储、报表等多个子系统，应拆成多条任务
- 如果请求本质是在修正 `aicore` 自身流程，应归到 `task-governance`

### 3.2 主入口优先

在 `approve` 前，必须明确：

- 主入口是什么
- 是否存在兼容入口
- 是否需要双改
- 为什么需要或不需要双改

如果主入口不明确，不允许批准任务。

### 3.3 修改范围冻结

在 `approve` 前，必须明确：

- 允许修改哪些文件
- 明确不应该改哪些区域
- 验收依据是什么
- 回退方案是什么

不要允许 Agent “顺手重构无关代码”。

## 4. 统一性契约

以下内容必须在任务级别显式确认，不能让 Agent 临场自由发挥：

- 时间语义：使用哪个时区，返回什么时间格式
- 错误语义：错误码是数字还是字符串，哪些错误属于哪一类
- 删除语义：软删除还是硬删除，字段名是什么
- 状态语义：状态机有哪些状态，转移规则是什么
- 输入输出字段：例如 `mobile` / `phone` 只能选一个稳定名称
- 分页语义：只能统一一种主格式
- 日志语义：新增日志应使用现有主日志体系
- 常量语义：同一业务常量不能散落在多个模块重复定义

如果这些语义没有写入 `brief` 或 `update` 信息，默认视为未完成任务级架构审核，不应进入 `approve`。

## 5. 减枝契约

Agent 不得为了“显得完整”主动扩展场景。

禁止行为：

- 明明只需要一个 API，却额外实现 3 到 4 个近似 API
- 产品没要求，就顺手抽象额外层级
- 为了未来可能用到而提前实现兼容分支
- 看到多个相似文件，就不经批准统一重构

允许行为：

- 只改当前任务明确要求的最小闭环
- 把未来问题写进 `risk`，不顺手实现

## 6. 命名与结构契约

以下现象都视为违规信号：

- 同一业务含义在不同地方用不同变量名
- 同一常量在多个文件重复定义
- 写了不用的函数、变量、类
- 写了重复功能的 endpoint / service / helper
- 返回 JSON 层级忽深忽浅
- 工具格式化函数分散重复实现

处理方式：

- 任务中若需要统一命名或结构，必须单独立项
- 当前任务未授权时，不自动扩大成“全项目风格治理”

## 7. 数据与持久化契约

`aicore` 不负责判断数据库设计优劣，但要求任务明确记录以下边界：

- 当前任务是否允许修改数据库结构
- 当前任务是否允许新增裸 SQL
- 当前任务是否允许修改事务行为
- 当前任务是否允许改动软删除/状态字段

如果这些边界未明确，默认禁止修改对应区域。

## 8. 测试与完成态契约

### 8.1 验收标准必须具体

不要写：

- “功能正常”
- “测试通过”
- “逻辑正确”

应该写：

- 哪个命令、接口、页面、测试用例能证明结果成立
- 哪些已知限制仍然存在
- 哪些风险本轮没有覆盖

### 8.2 完成不等于写完代码

以下都不算完成：

- 代码只改了一半
- 旧调用点没清干净
- 新旧命名混用
- 测试没更新
- 账本记录靠聊天记忆补写

只有在任务边界内的代码、验证、留痕、完成态都闭环后，才算完成。

## 9. 违规清单

下面这些行为应被视为 Vibe Coding 高风险信号：

- 未 `start` 就开始改代码
- 未 `approve` 就扩大实现范围
- 未 `log-write` 就继续下一轮修改
- 用临时脚本大规模替换但没有回退点
- 用“我顺手修了”扩大任务边界
- 发现问题后直接重写另一个模块而不拆任务
- 用模糊描述掩盖未完成状态

## 10. 推荐用法

起任务时优先补齐这些字段：

- `main_entrypoint`
- `allowed_files`
- `baseline_refs`
- `success_criteria`
- `risks`
- `review_summary`
- `rollback_plan`
- `dual_write_reason`

如果你不懂安全、架构、数据库细节，也没关系；`aicore` 的职责不是让你一次看懂所有坑，而是让你至少先把当前任务的架构约束、修改范围、验收方式和回退策略固定住。

这会显著降低“AI 看起来做了很多，实际上把系统改乱”的风险。
