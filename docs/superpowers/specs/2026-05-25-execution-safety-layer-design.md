# 执行安全层设计文档：改动留痕与完成态账本 MVP

## 1. 背景

现有 `aicore` MVP 已具备任务启动闸机能力，可以在进入执行前冻结任务边界、约束和审批状态。

但真实使用中还存在另一类高频风险：

- Agent 改动已经发生，但新版本没有被及时保存
- 后续错误回退或覆盖导致已完成修改丢失
- 某个细节已经生效，但没有被记录为系统当前事实
- 新会话无法可靠知道“现在系统到底已经具备什么能力”

因此需要新增一层独立于任务启动器的 `执行安全层`。

## 2. 目标

本 MVP 只解决两件事：

1. Agent 每次成功写文件后，都必须立即形成可恢复版本
2. 某个修改被确认已生效后，系统当前能力和限制必须被写入完成态账本

本层要解决的是：

- 产物丢失
- 回退误伤
- 完成态事实失忆

## 3. 范围

### 3.1 本版本负责

- 仅覆盖 Agent 产生的文件写入
- 为每次 Agent 写入生成结构化事件记录
- 为每次 Agent 写入保存文件级快照
- 为阶段性完成动作生成 checkpoint
- 维护一份系统完成态总账本
- 让账本条目可以追溯到任务、事件或 checkpoint

### 3.2 本版本不负责

- 监听用户所有手工改动
- 替代 Git
- 自动判断“模型是否降智”
- 自动从所有 diff 推导完成态事实
- 自动更新所有项目文档

## 4. 两层职责边界

### 4.1 版本留痕层

版本留痕层负责 `历史事实`，回答：

- 刚刚改了什么
- 改坏后如何恢复
- 某个文件上一版是什么
- 某一轮修改的稳定点在哪里

它不负责判断某个功能是否已经成为系统完成态。

### 4.2 完成态账本层

完成态账本层负责 `当前事实`，回答：

- 当前系统已经具备什么能力
- 哪些入口已经生效
- 哪些限制真实存在
- 哪些兼容性约束需要持续保留

它不记录每一次细粒度写入过程。

### 4.3 两层交接规则

数据流必须是单向的：

```text
Agent 写文件
  -> 自动生成事件和快照
  -> 人工确认变化已生效
  -> 更新完成态账本
```

原则：

- 每次写入都必须进入版本留痕层
- 只有完成态变化被确认后，才进入账本层
- 账本更新必须能追溯到具体任务、事件或 checkpoint

## 5. 版本留痕层设计

### 5.1 目标

只要 Agent 成功写入文件，本次修改就不能只存在工作区中，而必须立刻变成一个可恢复版本。

### 5.2 最小数据流

每次 Agent 成功写入后，执行以下步骤：

1. 生成一次写入事件
2. 保存涉及文件的快照
3. 追加结构化事件日志
4. 在显式阶段完成时生成 checkpoint

### 5.3 目录结构

```text
.aicore/
  history/
    events/
      2026-05-25T10-30-15Z-task-001-event-001.json
    snapshots/
      task-001/
        event-001/
          src__auth__login.ts
    checkpoints/
      task-001/
        checkpoint-001/
          manifest.json
```

设计意图：

- `events/` 用于查“发生过什么”
- `snapshots/` 用于查“恢复哪个文件版本”
- `checkpoints/` 用于查“回到哪一个阶段稳定点”

### 5.4 事件结构

第一版事件最小结构如下：

```json
{
  "event_id": "event-001",
  "task_id": "task-001",
  "session_id": "session-abc",
  "timestamp": "2026-05-25T10:30:15Z",
  "kind": "file_write",
  "files": [
    {
      "path": "src/auth/login.ts",
      "change_type": "modified",
      "snapshot_path": ".aicore/history/snapshots/task-001/event-001/src__auth__login.ts"
    }
  ],
  "summary": "更新 JWT 登录校验逻辑"
}
```

### 5.5 checkpoint 语义

checkpoint 不是替代细粒度快照，而是标记：

- 一轮相关修改已经完成
- 当前状态适合作为阶段恢复点

checkpoint 只能由显式动作触发，不能在每次写入后自动创建。

## 6. 完成态账本层设计

### 6.1 目标

账本负责记录：

- 当前已生效的能力
- 已建立的入口
- 当前真实限制
- 已知兼容性约束
- 需要持续记住的工程事实

### 6.2 文件形式

第一版使用单一总文档：

```text
.aicore/system-ledger.md
```

第一版不拆模块子账本。

### 6.3 文档结构

建议固定为以下区块：

1. `Current Capabilities`
2. `Entrypoints`
3. `Limits And Boundaries`
4. `Compatibility`
5. `Known Risks`

### 6.4 记录原则

账本只记录已确认生效的 `当前系统事实`，不记录：

- 每次小改动细节
- 临时实验
- 尚未确认生效的尝试
- 纯过程性说明

## 7. 账本更新触发规则

### 7.1 只留痕、不更新账本的情况

以下情况默认只保存快照和事件，不更新账本：

- 重构、重命名、拆文件等过程性改动
- 试验性改动
- 尚未确认生效的改动
- 不改变用户可感知能力或系统边界的纯修复

### 7.2 应更新账本的情况

只有以下情况才应更新账本：

- 新增一项已生效能力
- 某个入口正式建立、变更或下线
- 某个限制被确认或解除
- 某个高价值工程事实需要长期保留

### 7.3 完成态确认语义

账本更新不直接绑定“写文件成功”，而绑定：

> 某个变化已经被确认属于当前系统真相的一部分。

第一版仅允许人工确认触发账本更新。

## 8. CLI 扩展建议

为避免一开始做重，第一版建议只增加与现有 CLI 风格一致的最小命令：

1. `log-write`
   用于在 Agent 成功写文件后记录一次结构化写入事件并保存快照。

2. `checkpoint`
   用于在一轮相关修改完成后生成阶段 checkpoint。

3. `ledger-confirm`
   用于把某次任务/事件的完成态变化写入 `system-ledger.md`。

命令名后续可调整，但语义需要保留。

## 9. 与现有任务启动器的关系

任务启动器负责：

- 冻结任务边界
- 管理 `draft/reviewing/approved/rejected/superseded`

执行安全层负责：

- 保存 Agent 改动版本
- 固化系统完成态事实

两者关系如下：

```text
任务启动器定义“这次要改什么”
执行安全层记录“实际上改了什么”
完成态账本记录“现在系统已经会什么”
```

## 10. 成功标准

本 MVP 达成以下条件即可视为成功：

- Agent 每次写文件后都能生成事件和文件快照
- 某轮任务可显式生成 checkpoint
- 系统存在一份完成态总账本
- 账本更新只发生在人工确认之后
- 账本条目可追溯到任务、事件或 checkpoint
- 所有路径与文件操作设计不依赖 macOS 特性，兼容 Windows 与 Linux

## 11. 后续版本边界

以下内容留给后续版本：

- 监听全部工作区改动
- 自动识别模型降智
- 自动合并重复能力描述
- 基于事件日志自动生成大部分账本内容
- 可视化历史回放

## 12. 结论

执行安全层 MVP 的核心不是“让 Agent 更聪明”，而是：

> 让 Agent 每次写入都不会轻易丢，让系统每次完成态变化都不会轻易被忘。

它与任务启动器一起，分别补上：

- 开始前的边界稳定
- 执行中的版本安全
- 结束后的系统事实沉淀
